Effect Handlers

  • Strong foundations in category theory (Plotkin&Pretnar, 2009).

  • Algebraic effects are freely composable (unlike monads) and are typed. (unlike shift/reset, call/cc etc)

  • Moreover, they can be used to subsume many control flow constructs that are usually dealt with in an ad-hoc manner:

    • Exceptions.
    • Input/Output
    • State
    • Iteration (yield)
    • Asynchronous control (async/await)
    • Probabalistic programming

    read more at: https://​koka-​lang.​github.​io/​koka/​doc/​book.​html

Exceptions in C

Defining a new exception effect: 

DEFINE_EFFECT1(exn, raise)          // effect exn with operation raise

void exn_raise(const char* msg) {
  perform(OP(raise), msg);          // perform is a primitive function
}


We can already use it as: 

int safe_div( int x, int y ) 
{
  return (y!=0 ? x / y : exn_raise("divide by zero"));
}

Defining the handler in C

If we had nice syntax, we could imagine writing: 

try {
  safe_div( 42, 0 );
}
catch(exn) {
  raise(msg) { 
     printf("exception raised: %s\n", msg);
  }
}

but with our library we need to write it more explicitly..

Defining the handler in C

void* handle_exn_raise(resume_t* r, void* local, void* arg) 
{
  printf("exception raised: %s\n", (const char*)(arg));
  return NULL; 
}
const handlerdef_t exn_def = {    // a handler is like a v-table
  EFFECT(exn),
  { OP_NORESUME, OP(raise), &handle_exn_raise }
};
long handle_my_exn(long (*action)(long), long arg)
{
  return handle(&_exn_def, NULL, action, arg);    // handle is a primitive function
}

Applying the handler in C

void* body(void* x) {
  return safe_div( 42, x );
}

int main() {
  handle_my_exn( &body, 0);   // handles any exception anywhere in body
  return 0; 
}

“Strand local state” in C

DEFINE_EFFECT1(input,getstr)       // effect input with operation getstr

const char* getstr(void) {
  return (const char*)perform(OP(getstr),NULL);
}


And we can use the operations directly: 

void hello() {
  printf("hello %s\n", getstr());
}


Generally gives dynamically scoped read state: for example, we can implement HTML request reading functionality in a web server without having to plumb around a req object everywhere manually.

Handling with Resume

Our handler looks like: 

void* handle_input_getstr( resume_t* r, void* local, void* arg ) {
  return resume(r,local,"there");
}

It uses resume to resume back to the point where getstr was invoked and return with the result "there".

Asynchronous programming in NodeC

Asynchronous programming is good for interleaving computations that are I/O bound.

The libuv C library gives a portable async interface that is callback based.

This is very cumbersome to use though, so NodeJS gives an interface where you can use JavaScript instead:

  • NodeJS   ≡   libuv + V8 + JavaScript

But we can do the same with libhandler directly in C:

  • NodeC    ≡   libuv + libhandler + C

Asynchronous programming in C with LibUV

Very cumbersome as it is all callback based, and we need to encode all local state explicitly; (C++ async/await would make this state-machine encoding easier) 

void async_stat( const char* path  ) {
  uv_fs_t* req = (uv_fs_t*)malloc(sizeof(uv_fs_t));
  uv_stat(uv_default_loop(), req, path, async_stat2);  // register
  return; // return back to the LibUV event loop
}

void async_stat2( uv_fs_t* req ) {                     // callback
  if (req->err) ... // an error
  uv_state_t stat = req->statbuf;
  uv_fs_req_cleanup(req);
  free(req);
  ... ;   // continuation code
}

NodeC: Use LibUV in C with Effect Handlers

We define a single async effect with one operation, await: 

DEFINE_EFFECT1(async,await)    // effect async with operation await

int await( uv_req_t* req ) {
  return (int)perform(OP(await),req);
}

The handler for async just returns from await without resuming: allowing the libuv event handling loop to run. However, we store the resumption in the request object: 

void* handle_async_await( resume_t* r, void* local, void* arg ) 
{
  uv_req_t* req = arg;
  req->data = r;          // store resumption `r`
  return NULL; 
}

A generic callback

We now have a single callback function that just invokes the resumption stored in the request object: 

void async_callback( uv_req_t* req ) 
{
  resume_t* r = req->data;
  resume(r, req->result);    // and resume!
}

Wrappers for the LibUV API

Using the generic callback, we can now write wrappers around the primitive asynchronous API: 

int async_stat( const char* path, uv_stat_t* stat ) 
{
  uv_fs_t* req = (uv_fs_t*)malloc(sizeof(uv_fs_t));
  
  uv_stat(uv_default_loop(), req, path, async_callback);  // register
  int err = await((uv_req_t*)req);                        // and await
  
  *stat = req->statbuf;
  uv_fs_req_cleanup(req);
  free(req);
  return content; 
}

Lots of tedious work… but then…

Using asynchronous LibUV

We can directly call the asynchronous functions as if they are regular function calls: 

long read;
int err;
while (total < size && 
       (err = async_fread(file, &buf, -1, &read)) == 0 && 
       read > 0) 
{
  total += read;
  ...
}

this is why we need primitives in WASM to capture- and resume delimited computations.

Note: in contrast to async/await, there is no “color” and the async call can occur in regularly typed code (e.g. no transformation by the compiler is needed).

Multi-prompt delimited control is used by both libhandler and libmprompt as the primitive abstraction on which effect handlers are implemented.

Multi-prompt is Well-Typed

This works as each marker m is typed with the “answer” type: 

prompt :: (Marker a -> a) -> a 
yield  :: Marker a -> ((b -> a) -> a) -> b

note: this is not possible with shift/reset, call/cc, coroutines, fibers, etc.

Libmprompt: multi-prompt in C

// Types
typedef struct mp_prompt_s  mp_prompt_t;     // resumable prompts
typedef struct mp_resume_s  mp_resume_t;     // single-shot resume

// Function types
typedef void* (mp_start_fun_t)(mp_prompt_t*, void* arg); 
typedef void* (mp_yield_fun_t)(mp_resume_t*, void* arg);  

// Continue with `fun(p,arg)` under a fresh prompt `p`.
void* mp_prompt(mp_start_fun_t* fun, void* arg);

// Yield back up to a parent prompt `p` and run `fun(r,arg)` 
void* mp_yield(mp_prompt_t* p, mp_yield_fun_t* fun, void* arg);

// Resume back to the yield point with a result; can be used at most once.
void* mp_resume(mp_resume_t* resume, void* arg);
void* mp_resume_tail(mp_resume_t* resume, void* arg);
void  mp_resume_drop(mp_resume_t* resume);

This is the primitive interface that both libhandler and libmprompt provide (with a small library on top that give effect handlers).

Potential Implementations

With the multi-prompt interface, we abstracted over how stacks are represented and we open up the way for multiple implementation strategies. These include:

  • “split stacks”, used by multi-core OCaml and early implementations of Rust and Go.

  • Relocatable split stacks.

  • Fragment stacks (“fstacks”), used by libhandler.

  • In-place growable stacks (“gstacks”), used by libmprompt.

  • Fibers, provided on Windows.

  • Monadic, Async/Await, or CPS transformation.

  • ?

Split Stacks

Use small stacklets for the stack; every function checks if there is enough stack left and if needed allocates a fresh stacklet. On return, a function checks if it needs to return to the preceding stacklet.

  • Small memory footprint.
  • Addresses in the stack are stable (!)
  • Supported by LLVM and GCC.

But..

  • Impacts all function prologues and epilogues.
  • Suffers from the “hot-split” problem where a loop uses a function that crosses over to another stacklet which can impact performance.
  • Calling into C requires switching to the system stack.

Relocatable Split Stacks

As split-stacks, but instead of linking new stacklets, re-allocate a stacklet when more space is needed.

  • No more “hot-split” problem.

But..

  • Can be more expensive as relocation requires a linear copy.
  • Stack addresses are not stable !! This technique can only be applied if all pointers to the stack can be adjusted (so usually only languages with a GC use this).
  • Calling into C requires switching to the system stack.

Fragment Stacks (“fstacks”) (used by libhandler)

Use the regular stack, and capture- and restore fragments of the stack “in place” to maintain stable stack addresses.

  • Small memory footprint
  • No changes to the compiler or calling convention

But..

  • Addresses into the stack are stable but may be invalid while being swapped out – cannot address the stack safely across prompts.
  • Needs linear copy for both capture- and resume
  • Difficult to maintain good backtraces.

In-place Growable “gstacks” (used by libmprompt)

Use virtual memory to reserve address space for large (8MiB) stack segments and commit on-demand as the stack grows.

  • Addresses into the stack are stable.
  • Can grow in-place (no linear copies required).
  • No changes to the compiler or calling convention. Very robust.

But..

  • Needs a 64-bit system for enough virtual address space.
  • Uses a minimum of 4KiB per prompt handler.
  • On systems without overcommit (e.g. BSD, macOS) we need a signal handler to handle demand-paging.
