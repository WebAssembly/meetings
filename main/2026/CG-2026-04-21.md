#### [Simplifying JS-API test syncing](https://github.com/WebAssembly/spec/issues/2130) (Ryan Hunt, 15 minutes) 

RH: In my new role as a JS API editor, I'm trying to clean up some stuff. One tricky part of contributing to the JS API spec is the testing situation. There are tests that use the WPT harness. Right now the spec repo has a bunch of tests, the proposals add their own JS API tests. Canonical WPT also has tests. There is no automation for syncing any of these, and they have all diverged. It would nice to have automated syncing.

Proposed using upstream WPT as the canonical location for all tests. It supports "tentative" tests and everything could live in one place.

Alternatively we could use the testsuite repo that already merges all the core tests so it merges the JS API tests as well. Could then sync from that to WPT. We would want to add CI to proposals to actually run the tests. We also need a mechanism to warn on upstream contributions directly to WPT.

DeeptiG: Are you thinking of merging all the in-progress proposals into a single testsuite? Would they be tagged somehow so we could filter out tests for some proposals?

RH: Not sure, how is this handled for core tests today? They have a similar problem. In the testsuite repo we have directories for the proposals and they get mashed together?

KM: for the core ones, by the time it’s in there, you’re expected to implement them all. But for in progress ones you might not. For test262 there’s a comment at the top that is parsed that tells you what else you need to run it. It encodes the dependencies there.

RH: now with WPT, when things get to phase 3 people just start committing stuff, and it’s fine if browsers don’t pass. We could also group into folders and add comments but it’s expected that not everything will pass.

DG: yeah having them tagged or in a folder makes it easier to run or filter out, that would be nicer. The general benefit of having more tests is more important but if we can make them easier to use too, that would be great

RH: when we write API tests for a feature, having them in a subfolder would make it easier.

KM: Question about how harness changes will be picked up. If one repo modifies the harness and that breaks another proposal, then what? Maybe we need a policy that harness changes go to the main repo. Or git submodules or something?

RH: yeah there are things to figure out still. I would lean toward just having the harness is everyone is assumed to have the same one, and the forks could be updated. Hopefully it’s relatively slow-changing and not need a ton of extensions. But if that turns out to be a problem we can rethink it.

DG: you said adding tests when they get to phase 3. What about even having them at earlier phases? It’s nice to encourage people to write tests even earlier and it really helps for fuzzers.

RH: there’s no reason we couldn’t. The goal is just to make it much easier to write JS API tests by  making as much automatic as possible.

AR: I had assumed phase 4 would be where we merge things together, since features are more stable.

KM: but if you wait until 4, then implementations don’t have tests.

RH: to be clear, this would be just when things get merged into the testsuite repo.

BV: If we can still easily run tests within the proposal repos, I'm not sure I see the point of aggressively merging early-phase proposals into the common testsuite repo.

RH: tests that are only in spec proposals don’t get run until they go into WPT, so this would make it much easier for us. I think Google pulls directly from the spec

KM: we have a mix, it’s messy. It would be nice to have everything together


#### [Revisiting the "no overloading" design principle](https://github.com/WebAssembly/shared-everything-threads/pull/110#issuecomment-4216333083) (Thomas Lively, 20 minutes)

TL presenting [slides](https://docs.google.com/presentation/d/1ZMmsrpE4zMc8OR9oP-13VxqR2oXE5aoZbCM6MdLmS1E/edit?usp=sharing)

TL: The shared-everything proposal introduces a whole bunch of RMW operations, beyond linear memories. Each of these instructions operates on many value types. With the current proposal, we have one rmw.add instruction and just like struct.get it operates on different types. AR noted that because they operate on the values instead of just pushing things on the stack, it violates our no-overloading principle. For other instructions (e.g. add) we have different instructions for the different types. If we did that we’d have 57 new instructions instead of 24 (or 66 once you include shared types). 

CW: one difference wrt i32 and i64.add: wouldn’t overloading those violate our principal types property?

TL: true. We could relax our policy to allow type variables, but if we didn’t do that, then you’re right

CW: that makes me more comfortable with this than if we did that with e.g. add.

AR: with shared, we expect ref.eq to also work on shared, right? We view that as a parametric part of the comparison?

TL: I don’t see what the difference is though, because it could have a different bit representation

AR: to me the difference isn't so much whether it can have multiple reprenstations... I don’t know.

BT: why do we need atomics on non-shared references?

TL: we could discuss that, but it’s orthogonal. It follows the example of the linear memory atomics.

BT: generally overloading is a problem for interpreters, you have to pull in the value tags and makes it a global cost. So it’s not clear from this slide which way the disambiguation would have to go?

TL: for each one of these lines, each combination. So for struct.atomic.rmw.and would need 2 overloads, i32 and i64

BT: so youd have to look at the field type to see wehther it’s an i32 or i64 add?

TL: yes, just lke a struct.get

BT: yeah those are already not too fun

KM: don’t you have to look up the type anyway already?

BT: I think we cheat and rewrite struct.get in place to an internal opcode

TL: for v8 we're not doing in-place interpretation, fewer instructions would definitely better

KM: we do do in-place interpretation. I’d have to get back to you on what the consequences would be for that. If it’s the same as what we do for regular structs, it might not be so bad.

BT: for struct.get there’s 2 ways t make it faster: rewrite the instruction, or use  side-table entry. It’s a tradeoff. It’s not just an overloading thing, but overloading makes it worse, I think.

AR: I’d lke to raise the broader question of, if we go with overloading here, what does it mean? For now we have a very clear line that we don’t cross, that guides the design. If we erase that, i would like to see a new proposal for where we draw the line in the future so we don't go overboard. But I don’t know where that line would be. So i’d like to see what we’d do instead.

CW: for me it’s more about principal types. What were you thinking was the distinction?

AR: for get, you don’t have to look at the value, you just push it. For other operations, the operation is different depending on the type, you have to look at the value. That’s a clear rule, so i’d like to see a proposal for what the rule should be otherwise.

CW: [something about type hierarchies]

AR: for reference types, you just need the supertype.

[missed comment from TL]

AR: in cases where you don’t really care about the representation, you just pass it around

BT: for stack switching we implement contref as a fat pointer, so it’s already different

AR: yeah you don’t have to compare those.

TL: on what the rules should be, I agree with CW that as long as you don’t add new type variables into the principal types rule, it should be good. For where it “doesnt matter” what the type is for parametric cases... of course it does matter for real implementations. Would be interested to hear more about how it affects interpreters.

BT: we aren’t doing FP atomics?

CW: a separate discussion, I forget the rationale.

TL: for next steps, I’d love to get KM’s input on the interpreter question. We can of course add 66 new instructions if we have to, but I’d prefer not to.


#### [JS Text Encoding Builtins](https://github.com/WebAssembly/design/issues/1583) (daxpedda, 20 minutes)

DP presenting [HackMD](https://hackmd.io/90QpK3FCR0-q1AZQh_udKA)

This proposes 5 functions: one for decoding from GC arrays and one from linear memory, another for measuring in UTF-8 (in JS, the length is in WTF16, so we need a separate function). And one for encoding for GC arrays and for linear memory. the goal is a more performant way for toolchains to decode and encode strings from their native format to JS string (in the form of an externref). 
There are a bunch of open questions. One particular one is how to deal with wasm64. An easy solution but maybe controversial is to make everything an i64 instead of 32, e.g. the length. Then a user who doesn’t want to deal with bigger things can just not do that, and then not worry about trapping. But we need the i64 for encoding and decoding, to handle addresses that might be bigger than i32.

TL: for the i32/i64 question, we are planning to add precedent for overloading the builtin imports, e.g. for string builtings for shared strings is to just allow them to be imported with 2 different function tuypes. E.g. today’s and one with shared externrefs. So for this you could import with either i32 or i64 in the signature, and it would expect different kinds of memory. So that would be consistent.

DP: i would argue that this is hopefully unnecessary. Even if you’re not using 64 bits you can still deal with 64 bit addresses and just convert

AR: the other option is to have 2 names. One thing that worries me is that the component model is going the opposite direction and disallowing importing 2 things with the same name. So far we aren’t having overloading on imports, but maybe we don’t want to start now.

DP: i would just change everything to i64, it would work with both. If that’s deemed not good enough then we can figure out what to do.

TL: from a toolchain perspective, i would have a weak preference for overloading, but it’s easy enough to work around.
What do you see as the next steps here?

DP: I believe I have to request a phase 1 proposal. I’m here to do that now.

CW: we can do a consensus for that. Be aware that for future phase advancements, you should add your intention to advance in the agenda ahead of time.

CW: any thoughts on this? Maybe RH?

RH: overall it looks reasonable. I have ideas we can discuss later. I’m conflicted on the 64 bit thing, there are merits to both. Having them all as i64 is annoying. Also having multiple results is annoying especially when calling builtins. We’ll have to support both anyway. No great ideas right now.

JK (chat): in practice, browsers cap JS strings at somewhere around 2**28 to 2**30. I expect that to remain the case for a loooong time, because to bump it you'd have to make all strings bigger, and given how common strings are (and how many of them are short) that'd be hugely expensive for all existing code/websites out there.




GB: one decision here: the problem of how to reference the memory, when you encode into linear memory, how to reference it. The current design uses an externref to represent the wasm memory. It would be good to get feedback on that for future phases.

RH: I thought about that; it’s nice but i have concern that that could be slow, it’s not been optimized to go through the WA.memory object. Maybe we could optiimize it; if so it would be nice.

AR: what are the other options?

RH: one would be to just hardcode to memory 0; not great. Or name mangle. There was another one, I forget

AR: anything that ties it to a specific memory index or name will just break tools like module mergeing. You never make indices or names visible outside.

TL: you could use export names, that would be robust to merges and splits.

RH: or have a dedicated slice type that refers to memory or GC array, or a memory ref. But those are bigger changes with tradeoffs.

DP: what would be slow? Getting the externref or passing it?

RH: you need to make sure the ref is an object, then cast to WA.memory. Then you have to find the most current data pointer and bounds length. Since WA.memory wraps an array buffer thats a couple indirections. Maybe you could cache, i’m not sure. But the naive one would be multipel casts and indirections.

DP: but I would imagine that any way to get these would have those downsides.

RH: yeah the casting is inherent if you use an externref. Some schemes would bind the memory directly so there’s no casting.

SD: if we add core conversion from GC arrays to linear memory and back, core insts would have internal access. If we could create GC arrays from linear memory content, you could build the linear memory version of the builtins using the GC versions. You could locally recognize that the GC array doesn’t actually need to be allocated so it could be easier to make that fast than the externref-memory one. 

DP: the main concern is that we want to avoid copying. You did argue that that could be optimized, but it would be preferable not to have to rely on that optimization. From the toolchain perspective, if we assume that e.g. rust one day supports all of the GC stuff, the only way currently to do that is inline asm which is unstable but those instructions might not always be right next to each other, so that could hinder the optimization. But it seems less than ideal to rely on it.

ML (chat): It can't be optimized away, Wasm arrays have a maximum length. So creating a Wasm array of a wasm memory needs to trap if the Wasm memory is too large.


KM( chat): I think the peephole optimization would be hard to do for an inplace interpreter. Maybe Ben has thoughts too?

TL (chat): Matthias, it would be just a slice of a memory. We have precedent for assuming the intermediate arrays can be optimized out in the configureAll builtin in the custom descriptors proposal.


CW: Unanimous consent poll for phase 1. Any objections?

No objection, proposal goes to phase 1. We’ll get the GH repo to set it up.
