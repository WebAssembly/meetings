#### 2.  JS Builtin naming conventions (from https://github.com/WebAssembly/esm-integration/issues/118#issuecomment-4290555584) (Guy Bedford, 5 minutes)

GB Presenting [Gist](https://gist.github.com/guybedford/1d19cc2f986b52d761ee732e729257d5)

RH: background is that we did JS string builtins, there were some discussion about naming but maybe not enough, it was hard to foresee what would come up in the future. Now there’s custom descriptors, string constants, text encoding, etc. It would be good to be on the same page on the scheme and have it be future proof. On the scheme point and JS prefix, I’ve come to prefer the wasm: scheme instead of wasm-js. It’s nicer to have just one scheme to worry about. I think JS will be the only category of builtins. I wouldn’t want to register e.g. wasm-component, wasm-node etc with IANA. Let’s just treat them like URLs with paths. There is just one backwards compatibility to preserve, js-string builtins. Haven’t thought a lot about the casing rules.

TL: Would you recommend for the Custom Descriptors builtin wasm:js-protoypes to switch to wasm:js/prototypes?

RH: yes, I think that would be good to put JS in a path-like scheme like that

AR: this generally makes sense. My one pet peeve with the JS world and URLs is that they violate the URI definition by misusing relative path, e.g. ./something is something. So use /something which would make it an absolute URL and not a relative URL.

RH: will have to think about that, I'm not super familiar with URL vs URI. IIRC WHATWG guidance is to ignore URI. I don’t want us to have to do URL normalization. Haven’t thought fully through it but it would be nice to do exact string matches.

AR: that would also suggest absolute paths rather than relative paths that have to be resolved.

TL: reminds me a lot of our treatment with UTF8 where we don’t do any of the recommended stuff. Maybe we could do something similar here

AR: not sure what you mean?

TL: we don’t do normalization etc

AR: that’s fine but since we’re picking syntax which is clearly URI with a scheme and path, it wouldn’t hurt to pick something that just normalizes to itself as a URI and don't diverge unnecessarily.

RH: I’m open to doing something like that but I’d want to see it written down, what it would look like and what it would parse to.

AR: the only difference is a slash after the colon

DP: if we’re already renaming, would we go with wasm at the start, that’s superfluous since we are already in wasm. Maybe something like builtins/ ?

RH: the scheme part came from a long discussion... the question was whether to register with IANA which we ended up doing. I lke having wasm as an unambiguous namespace. Would we be sharing the namespace if someone else wanted builtin/ ?

DP: I think it makes sense if we already have it registered.

GB: On the relative path topic, the exception that AR brought up, you’re referring to the HTML semantics of relative paths? In that spec, those semantics are specifically based on the entire string starting with the . followed by / in the filtering. So there’s a separate codepath for if the string parses as a URL, which I guess these would

AR: relative paths are also URLs. most of those pieces are also optional including the scheme. But some specs break the meaning of paths.

GB: effectively without a separate argument.

JK(chat): does the colon serve a purpose? Back when the IANA registration question first came up, it sounded a bit like we regretted that "wasm:js-string" accidentally looked like a URL scheme.

GB: also:  should these guidelines be written up?

RH: yeah the convention and why we have it should be somewhere even if not hard and fast. The textencoder rules also have that. And how they interact with e.g. WHATWG

GB: do you have suggestions on how to follow up on that?

RH: it's scattered around several repos. Maybe I can summarize things in an issue on the design repo, and the relevant parties can discuss there.

GB: yeah that would be good, so we don’t diverge.
The js-string can enable both paths, for backwards compatibility. The rest can change without needing a migration if we move fast enough. The one complexity is that builtins is no longer 1:1 from the builtin to the base that it enables, but maybe that’s OK.

RH: AFAIK the only place with real web compat restrictions is the string builtins. The rest aren’t shipped yet. So it’s a good time now.

GB: also we could potentially enable a simpler builtins JS, it could be a pain to add every builtin individually. If we just had a base JS to turn on it could encapsulate it nicely. For followups, I agree with the framing of builtins as matching, if we can get the guideline updated, that would be the main takeaway.

GB: since this is taking longer than planned, I’ll defer the presentation on deferred imports until next time.


TL: that sounds good. I like the builtins-js shortcut with all the JS builtins that would be nice. We could put text encoding and decoding there. We should figure out the extra "/" at the beginning sooner rather than later, to avoid extra implementations for custom descriptors.

GB: one last topic is the new builtin suggestion: WebAssembly.JSTag. I'ts a very useful JS builtin because of how self-contained its usage can be in Wasm.

RH: from a process perspective I imageine we wouldn’t need a full proposal, maybe its minor enough. Havent thought through it fully; its sort of a new case for builtins where it’s not so much about adapting something that you could have ipmorted, but getting it automatically imported. For string builtins, we had a clear scope, this is slightly out of scope, but seems reasonable. Maybe there’s a way to frame this where we don’t start having to allow everything to be imported.

GB: the question is more that if there is any interest, what’s the right way to go about it? Is it its own proposal, or just an issue?

RH: there is precedent for smaller things as issues.

GB: I’ll open an issue.

SD (chat): What is the main benefit compared to defining a one-line .js file next to the .wasm? Most builtins have a performance/access rationale, but here I'm not sure.
