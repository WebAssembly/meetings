## `wasi:http`: request and response should have a tracing identifier

Bailey: we've had some concerns about wasi:http and how it intersects with instance reuse. We should talk about that. Alex?

Alex: Pat identified issues wrt concurrent instance reuse with wasip3. Wasip3 adds the ability to send and interleave concurrent requests (note: hope I got that right).

Alex: right now there is a 1:1 correspondence between requests and handlers. That breaks now, and Pat showed an example with the CDN loop header running into problems.

Alex: general problem is that if everyone supports handling multiple requests per instance and supports concurrent processing then we're in trouble. Key insight: this requires the guest to not be malicious and try and mess things up.

Alex: host should be able to enable instance reuse without concurrent instance reuse. That would be the same world that we are in today.

Alex: say that a host wants concurrent instance reuse and wants to correlate in/out, the guest has to participate. That rules out malicious/untrusted guests because it depends on coordination.

Alex: option 1: strong relation between imports and exports based on the call stack. As of an hour ago: we realized that this might be hard to express in certain languages. For example: in Rust this should generally Just Work that if you call a request it'll work. But Till pointed out that in JavaScript would have trouble ("soup of stuff"). Joel pointed out Go right now is similar too.

Alex: we're ratcheting down. Say a host is using concurrent instance reuse in a language that has a soup of stuff. We might be able to add additional intrinsics to the component model to track when it switches what work it handles.

Bailey: Are proposals on the table to change the wit specs to solve this?

Pat: While we could solve individual cases that way, we would not solve the general case that way.

Bailey: the options I'm hearing here are: host magic and component model instrinsics.

Alex: wouldn't call it magic. Host has a notion of an async call stack. The idea of an insrinsic route is that we could embed this in the runtime. Do it once for the component in the runtime. Guests would be able to do additional things to help make it more accurate.

Bailey: would the intrinsics be additive? Did you get the chance yet to make a design sketch?

Alex: it's too fresh out of the oven (1hr ago), not yet.

Alex (in chat): Also FWIW I was figuring this would be the forum to discuss the viability of these intrinsics, and assuming consensus here I’d go through the rigmarole of writing up issues and such.

Till: the intrinsic is the only thing we can add that would help any with this. the WIT changes wouldn't help with this. you can always add the wrong thing. what the WIT changes would do is to help force a decision, but what we're really talking about is about runtimes. there are not many of us who need to care about it. given the intrinsic it should always be ergonomic and idiomatic to do the right thing inside the language runtime. this allows the right people who care about this to do the right thing over time.

Victor: Follow-on from earlier discussion. There is runtime support for which ever languages will support it out of the box right now. Then there is the idea of popular idea of separating out the change with intrinsics. Is it enough for wasmtime to expose this stuff that is already there and much easier to expose. Then do the work in 0.3.x later for the guest-side and more robust version of this feature.

Pat: So right now if wasmtime, if it makes it possible for the wasi-http implementation and possible for the runtime to observe, just not in the public of wasmtime today, then we could do that, then allow this for certain guests that have been written compliantly. We do not have a way to know ahead of time. This makes it dangerous to accept this. It will come up in weird cases where we will have to trap them. Concurrency is going to trap due to incorrect interleavings and puzzling behavior and bugs. The way I'm looking at now, that there is this category of problem of concurrent instance re-use, and once we make it really clear to write any guest in any guest language, then that is something we can enable for all component model async. Concurrent instance re-use does not look ready. A lot of the rest of wasip3 and component model async appears to be ready.

Daniel: I want to raise the issue that the http implementations all rely on polling-based interfaces. Is there a plan for completion-based async primitives within WASI as well and how will that change things? io_uring has been around for about 7 years now so it's not exactly new. How will that affect designs we're making now?
