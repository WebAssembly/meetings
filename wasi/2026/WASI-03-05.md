## Writing detailed WASI specifications using `specwit` (@tschneidereit)

till: showing off something I've built, explain how to use this overall and requirements.

till: why specwit? - WASI proposals currently exist of WIT interfaces + explainers. Looking at just `wasi:http` the entire interface is defined in two files. One for the world, the second - much bigger one - defines all the types.

till: the docs present in the WIT are a mix of doc comments and requirements for implementation. Implementation requirements which you would typically not see in doc comments.

till: we have a test suite for wasip2 and wasip3 which over the past few months have fleshed out by Igalia quite substantially which has lead to issues being opened on the implementation.

till: right now you have to make decisions on everything which doesn't just fall out of the type signature. When in doubt look at existing implementations and match their behavior. The wasm test suite does help with that. But the test files, while good, are quite long (multiple hundreds of lines for http)

till: if we compare the fetch spec, that's its own standalone document. It also includes things like normative references, which are things which the spec itself doesn't define.

till: there are reasons why we don't need to see the same level of detail as the web specs. But we should try and pursue the same general approach. We need a detailed description of what implementers need to implement to have a conformant impl.

till: the fetch spec has sections containing conformant algorithms which explain how to perform certain operations, which can directly be linked to. In the spec these are ordered sequences of operations that a conformant implementation must follow.

till: how do we bring this to WASI? I've tried to write specs for wasm over the years and bounced off. Writing specs is hard.

till: it would be great if we as a community could agree on a shared format for how to write specs and how to show them/render them.

till: so! I've built a tool for this! that's what `specwit` is. It can render doc-comments and interfaces.

till: but what I'm really excited for is the editor. It scaffolds out an HTML page which can be edited to e.g. specify the algorithms. It has support for links which can be referenced.

till: it also includes a list of metadata which is generated and can be used to link to external specifications. It includes links to renderers of those specs.

till: my goal is to remove as much of the boilerplate of writing specs as possible and handle it with the tool instead.

till: this generates output which e.g. respec and other spec tools can parse and work with. That means we can integrate into the existing spec ecosystem and become a publisher that creates specifications.

till: This is just a demo now, no vote. But I do want to eventually bring this to a vote.

till: On the WASI repo I'd like to host the spec.md, specwit.toml, together with the wit files. Then host the editor on something like `specs.wasi.dev`. The editor has support for snapshots and zip downloads?

till: assuming people agree that this makes sense, the other question is: what do specs mean for phase advancement criteria? We could have similar kinds of entrance criteria as the main wasi spec.

till: big downside is that now advancing a proposal becomes a lot more work than it was before. but in my opinion that kind of work is unavoidable as we aim for interoperability. My experience from work on the JS specs reflects that. The longer we wait with this kind of thing the harder it will be to fix interop issues and the harder it will be to fix those incompatibilities.

till: the sooner we tackle this, imo the better it will be for us.


yosh: huge fan of the approach. The sooner we do this work, the easier the work will be. Strong in favor of that. I have a general question that wasn't clear from the presentation? What are the files that are checked into disk? spec.md and a toml file?

till: only those two. If you go to my repo for specwit, `/docs/<github url>` it will look for wit files in that. It will look for spec.md and toml files. It will auto discover deps, based on the deps.toml file. Then it will save a snapshot.

till: The wit file is the source of truth for the API.. This is actually different from bikeshed and others. I wanted to have the idl file stay the source of truth and have the spec be in sync with that. Now the editor will automatically extend. What I am envisioning is that we would host just those two files and nothing more.

yosh: Have you given any thought on running the identifiers through the wasi test suite?

till: tracey is a tool yosh shared with me that I am very excited about. Ultimately there are two things that I would love to do with this, but I don't think should block initial adoption, formalizing the algorithim in the implementation language to make it machinable. Then have a checker tell you if you deviate from that format. That is what the wasm core spec has, e.g. the spec is executable. The other thing is being able to say here are the parts of the spec that are checked over here in the testsuite then an implementation can validate that it hits those cases. That would be a substantial advancement over all webspecs right now. If I or somebody else gets to that point, we should talk about it again.

Alex: One thought that I would have on the spec requirement for advancement, we should flush this out for our core ones cli, http, clocks, etc. Do this before we require it for others. I myself have no idea how we'd specify the monotonic-clock, so we will want to make sure we have that in place, then make it required for future proposals.

Till: Yes in the future I want to make a formal proposal for adopting this. We will have to find where the scope bottoms out. We can't be the ground truth for every part of the behavior. Where those boundaries are, we don't yet know and will find in pursuit of writing specs.
