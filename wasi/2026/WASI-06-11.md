## WASI 0.3.0 vote

[slides](https://github.com/WebAssembly/meetings/pull/2060/changes#diff-9903c9022c6618cb26d81d0658fcda8529c32e0c4bccee811a0f0da5f2d90c03)

**Bailey**: Just one agenda item today: running a concensus vote to launch wasip3. We're going to follow the same process we followed for WASI 0.2 in January of 2024. We're doing the same thing here. If we choose to launch we'll make additive changes-only in a patch releases on a release train. The last draft release was 2026-03-15. We're going to launch the next revision of phase 3 proposals.

Bailey: to be included in wasip3 a proposal must have reached phase 3 in the phase process, satisfied its portabily criteria, and be voted for inclusion by the SG. To meet the criteria we have a full feature description available, WIT available, and all deps are phase 3 too. For all of these we have been in our implementatio phase for a long time.

Bailey: we're going to run a consensus proposal: type SF, F, etc. in chat to vote.

Alex: Wasmtime is currently passing all tests in the wasi testsuite repository. These tests are similar to the wasip2 tests. Wasmtime has additional tests that cover edge cases around the component model. The coverage for WASI is really good though. For wasip2 we didn't have a full test suite, we used adapters on top of wasip1. Wasip3 tests are from the ground up to exercise wasip3-only functionality.

Bailey: the test run shows results from both wasip1 + wasip3. Victor can you speak to Jco?

Victor: Igalia integrated Jco with the new testsuite. Tomasz helped bring these tests over the finish line. We have good test coverage on the WASI side; we're currently at parity wrt Wasmtime on passing the WASI test suite (40 tests in the screenshot).

Bailey: we're going to first look at wasi-cli to advance to p3. We're going to say we're happy with wasi-cli, then wasi-http. When we are happy with both worlds, we are ready to launch p3.

Bailey: wasi-cli is meeting the criteria and passing all tests. Before we hold the vote, and questions or concerns? - no. Let's hold the vote!

- SF: 14
- F: 2
- N: 0
- A: 0
- SA: 0

Bailey: We can consider wasi:cli at phase 3. Next is wasi-http. It meets the criteria. I will call out that it must have at least two complete implementations, and demonstrate embeddability in production server contexts. Both Akamai and Cosmonic have embeddings.

Dave (in chat): does http depend on sockets?

Pat: it is a transitive dependency via `wasi:cli` as a dependency, but the types are not used.

Bailey: striking `filesystem` from the dep list as well for the same reasons. Let's move to vote:

- SF: 13
- F: 2
- N: 0
- A: 0
- SA: 0

Bailey: that is consensus, wasi-http is now phase 3. Now the next vote is to launch wasip3. The criterium is that there are two independent proposals which define worlds at phase 3. So with that let's move to a vote:

- SF: 15
- F: 0
- N: 0
- A: 0
- SA: 0

Bailey: and with that WASIp3 has launched! Congratulations everyone.

Bailey: we have draft release PRs in flight to release p3 release. Yosh and I will run the release for this.

Alex: my plan for wasmtime is to pull this into the main branch for wasmtime 47 to release sometime in July. Wasmtime 46 will have the wits pulled in and un-gated for June (22?). We are waiting for changes to fully land in LLVM. This is necessary for the Rust and C toolchains.

Bailey: it's a launch, doesn't mean it's done. We also have a release process that allows us to make more changes over time.
