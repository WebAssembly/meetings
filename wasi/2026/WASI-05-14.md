Bailey: Will you share a little info some of the other LLM produced implementations?

We can more easily apply and absorb the AI approaches and generated code bases. Where these repos often have a question mark is that they just haven’t pulled down all of these test components. Obviously with these LLM’s many already have, but some haven’t completely yet, but very possible and easy to do. 


### Open Discussion on WASIP3

**Bailey**: Now that we’re getting close to launching P3, we’re going to hold an open forum/discussion around P3. 


**Bailey**: One thing I want to callout is that there is a [new blog](https://bytecodealliance.org/articles/how-wasm-components-enable-pluggable-middleware) on the bytecodealliance.org from Elizabeth Gilbert that walks through a number of interesting concepts like service chaining and component composition, including a demo with wasip3 http middleware. 

**Pat Hickey**: I have a question. I’ve seen there are a lot of discussions about instance reuse for WASI HTTP in P3. Can someone summarize what the conclusion is? What is the story, backpressure, concerns, etc?

**Joel Dice**: The intention is that unlike with P2, with WASI HTTP in particular, the expectation is that instances will be re-used if they are not pushing back without back pressure. Whereas with P2, the expectation that we’ve created for component authors is that instances will not be reused. We wouldn’t do this in a point release of WASI because that would be very surprising but in the P3 release, we felt this would be the ideal time to change those expectations. 

**Joel Dice**: What power does the component author have to push back against instance reuse. In the p2 era we had serial instance re-use. In p3 we have concurrent instance re-use. As a component author you might want to pushback against either of these. In the current state of the record, is that you can use back-pressure. Incr and Decr intrinsics in the component model might be exposed in different ways used as functions. Until this number goes back down to user, I don’t want you to make any new calls into me. I don’t want any new concurrent calls. That’s the first aspect. There is also the very familiar wasi:cli/exit or exit-with-code. You presumably will use this with backpressure and I won’t do anything new and then I would call exit. This is how it can say I’m done. One idea that we’re thinking about is turning that into a component model intrinsic. 

**Colin Murphy**: Does this have anything to do with what I’m about to say. There is always this initialization time where I have a stream of bytes aka a stream of images. Is this going to allow me to use this same component on a bunch of requests? So I have this initialization time, then on the next time, I don’t have to do that initialization time to get started. 

**Luke Wagner**: This is one of the session instance re-uses. You get to reuse the memory that you setup. 

**Bailey Hayes**: What is the difference between instance re-use and a long-lived instances?

**Luke Wagner**: Multiple requests can be sent to the same Node.js process. It can be re-used a number of times. If you are generating one of the components, you may have to anticipate that you have to handle this scenario. If I emit a synchronous function, then the bindings have to do the back-pressure for you. This is not the same as a traditional process where it sits in a socket loop, it can be still serverless.
