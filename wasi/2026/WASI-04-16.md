### wasi-spi phase 2

**Lukas Barragan Torres**: As part of my masters at Ghent university working on SPI for WASI. SPI is a sync communication protocol used in embedded environments. Terms used in the protocol include:

MOSI: Master-Out, Slave-In
MISO: Master-In, Slave-Out

Lukas: our goal is to mirror the existing Rust embedded-HAL (Hardware Abstraction Layer) embedded interface as much as possible. However the existing interface makes a lot of use of in-place operations.

Lukas: our error variant is the same as the HAL variant but we added an extra string variant to capture “other” cases. For our operation enum we can’t directly write into buffers so instead we are passing an owned list<u8> as part of our read and write operations.

Lukas: we also added an operation result which isn’t present in the HAL interface. We did this because we can’t mutate through pointers, so instead we submit an operation and get a result out on the other side when the operation completes.

Lukas: We did remove the HAL “transfer in-place” function because we can’t directly write to the guest memory.

Lukas: To get resources we added an open function which takes an identifier.

Lukas: We have an implementation based on the embedded HAL. The only real logic we have is switching between the WASI and embedded HAL models. We created two host implementations to show that it works: Raspberry Pi 4 and Raspberry Pi Pico 2. Both implementations run on Wasmtime.

Merlijn: this is running on WASIp2 on the rpi pico 2.

Bailey: questions for Lukas?

Jeremy Fleitz: is there any concern if we do anything asynchronously with bus selection on SPI?

Lukas: SPI is synchronous-only.

Jeremy: I understand. But could the same WASI host write to two peripherals concurrently?

Lukas: because both peripherals are on the same bus trying to do this would result in garbled data.

Jeremy: I understand - so we would need some locking to ensure we don’t try and multiplex over the same bus. But what if we had multiple buses?

Lukas: the WASI SPI device handles locking the bus down when a transaction is ongoing. This is inherent for an SPI device.

Jeremy: that makes sense. So the transaction will lock the device while live. 

Merlijn: this is very similar to I2C - we abstract the lower level parts away and ensure that the host does this

Bailey: we should document this as part of the wit. I’d like this documented

Lukas: good point

Bailey: you could file an issue for that - that is a core part of the spec and we should document it.

Lann: I’d like for us to provide guidance on common chips/hosts/formats.Take a look at Linux API for busindex/chipindex,...

Bailey: would we want an identifier for this as part of the interface?

Lann: I probably would like that to be handled out of band.

Bailey: the proposal is missing some of the necessary advancement criteria covered that are needed for a vote. Can we push the vote to the next meeting to give you the chance to fill those out?

Lukas: yes that’s good

### Proposal to advance `wasi-cli: exit with code` to phase 3 (Joel Dice)

Joel: this is a real quick presentation to remove the exit-with-code feature gate. The existing exit only has a boolean representation for whether an exit succeeded or failed. Existing programs in the wild which we’d like to port to WASI have more than two exit statuses. This came up about 18 months ago in the context of C# when we were adding WASIp2 support there. The test suite depended on exit codes for specific tests.
Joel: the way the test runner communicated with the tests was via exit codes. Bottom line: there is existing software which depends on this. Jco has had it soon after introduced, Wasmtime as well. this has been fairly uncontroversial.

Bailey: questions for Joel?

Bailey: ok none it seems. here are the portability criteria: https://github.com/WebAssembly/WASI/blob/main/CONTRIBUTING.md#3-implementation-phase-wasi-subgroup

Bailey: we are not aware of any problems with the entry criteria. With that I’d like to hold a full vote.

SF: 12
F: 0
N: 0
A: 0
SA: 0

Bailey: with unanimous strongly in favor votes, it passes! Congratulations champions! - Operationally the champions can now go ahead and ungate the feature.

## Update on WASIp3 support for Jco (Victor Adossi)

Victor: support for p3 async is something I’ve been working on for the past 5 months, for Jco in particular. This includes semantics, streams, futures, etc in the JS host. I want to demo where we are now.

Victor: until now we had the semantics, implementing and testing it. Streams and futures have now been implemented as well and we’re close to finishing that up. Going to be working with Tomasz on that.

Victor: a new version of JS component bindgen is now available (as of 10 mins ago!) - the version of Jco which will be released is 17.7. I will be showing this off now.

Victor: The transpile function now takes an “asyncMode: ‘jspi’” argument. If you’re running this in Node.js you have to add certain flags. The Jco library gives you object files which you can write to wherever you want.

Victor: Jco provides a WASI shim for WASIp2. But we’re going to be releasing a shim for WASIp3.

Victor: We took some time recently to ensure that stream support for Jco was built not against readable/writable streams but (async) iterators instead. These are more flexible and generalizable. The exact shape of the objects that you get back from the streams are not well documented yet.

Victor: for our demo I will show a function which takes a list of streams and returns a stream of strings. What the mapping of this function becomes will be language-dependent.

Victor: <showing demo passing a JS generator function to an instantiated Wasm Component>

Victor: what’s new today is support for futures - which in JS are promises. In JS anything that implements a .then method (“thenable”) can be used as a future, which gives us a lot of flexibility in what we can pass.

Victor: <showing a demo of a function taking a JS string and returning a JS promise which when awaited returns the string>

Victor: all that’s missing now is one more quick test for futures. And then we will be ready to bring everything together and hopefully be able to ship WASI 0.3 shortly after.

Victor: This is in Jco main right now. It will all be part of Jco 17.7 which will be available shortly after this meeting (!!!). We don’t expect to break anything from now on. Please try it out and file bug reports!
